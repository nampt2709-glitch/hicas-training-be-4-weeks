A. SƠ ĐỒ TỔNG THỂ THƯ MỤC
--------------------------------------------------------------------------------

  LogAnalyzer/                    (thư mục project)
  ├── LogAnalyzer.csproj         ; khai báo .NET, gói Microsoft.Extensions.DependencyInjection
  ├── Program.cs                 ; điểm vào, đăng ký DI, vòng lặp menu console
  ├── Interfaces/                 ; hợp đồng (interface) — tên cũ: Abstractions/ — KHÔNG chứa thân hàm nghiệp vụ
  │   ├── IFileReader.cs
  │   ├── IResultWriter.cs
  │   ├── ILogGenerator.cs
  │   └── IBenchmarkRunner.cs
  ├── Services/                   ; triển khai hỢP ĐỒng — nối ứng dụng với file / quy trình
  │   ├── FileReaderService.cs
  │   ├── ResultWriterService.cs
  │   ├── LogGeneratorService.cs
  │   └── BenchmarkRunnerService.cs
  ├── Core/                       ; mô hình dữ liệu + thuật toán thuần (ít phụ thuộc cách chạy app)
  │   ├── AnalysisModels.cs
  │   ├── Analyzer.cs
  │   ├── Counter.cs
  │   └── GlobalExceptionHandling.cs
  └── Logics/                      ; tài liệu .txt (không tham gia build), giải thích project

  (Thư mục bin, obj: do build sinh — không cần chỉnh sửa tay.)

--------------------------------------------------------------------------------
B. TỪNG THƯ MỤC (MỤC ĐÍCH VÀ “BIÊN” TRÁCH NHIỆM)
--------------------------------------------------------------------------------

1) Thư mục Interfaces/  (trước đây tên Abstractions/ — tên mới: “Giao diện hợp đồng”)
   - MỤC ĐÍCH: khai báo hành vi mà chương trình cần (đọc file, ghi kết quả, sinh log, chạy benchmark)
     dưới dạng interface C#. Không đặt thân hàm nghiệp vụ ở đây.
   - CÁCH HOẠT ĐỘNG: mỗi file .cs chứa MỘT interface, namespace chung: LogAnalyzer. Các lớp
     trong Services/ “implements” các interface này. Program/ DI chỉ phụ thuộc tên interface,
     không phụ thuộc lớp cụ thể — dễ thay bằng lớp giả (mock) khi học test.
   - KHÔNG PHẢI: không chứa logic đếm, regex, thư mục ghi file cụ thể; đó thuộc Core hoặc Services.

2) Thư mục Services/
   - MỤC ĐÍCH: cài đặt các interface: “ứng dụng làm gì với bên ngoài” (đĩa, đường dẫn, ghép
     bước benchmark). Đây là lớp mỏng ỨNG DỤNG (application), không cần gọi trực tiếp
     File.* hoặc ghép 6 bước ở Program.
   - QUAN HỆ VỚI Core: Service **không tách hẳn** khỏi Core; **phải dùng** Core — xem mục D.
     Service không thay thế Core; nó gọi Core (Analyzer, Counter, ErrorCatalog, …) sau khi
     đã có dữ liệu (ví dụ: chuỗi từ IFileReader, hoặc từ chuỗi sinh từ LogGenerator nội bộ trước
     khi thống kê bằng analyzer — Luồng thực tế: BenchmarkRunnerService đọc qua IFileReader
     rồi dùng Analyzer + Counter; ResultWriterService chỉ nhận BenchmarkReport đã tính sẵn).
   - FileReaderService / ResultWriterService: gần với “infrastructure / adapter với hệ thống tệp”.
   - BenchmarkRunnerService: “orchestrator” 6 lần đo — **có mục đích riêng**: gom thứ tự Sync/Async
     và 3 cách đếm; nó gọi Core, không nên nhét 6 bước này vào Program (tránh dài, khó test).

3) Thư mục Core/
   - MỤC ĐÍCH: tập trung thứ “ổn định” theo bài toán: mô hình dữ liệu kết quả, tách token, đếm
     tần suất, danh mục lỗi, xử lý exception toàn cục. “Core” ở đây nghĩa là NỘI LÔNG
     phân tích, không phải tên .NET Core runtime.
   - CÁC LỚP: phần lớn static (Counter, Analyzer, GlobalExceptionHandling) vì bài tập/ demo ưu tiên
     rõ ràng, không cần nhiều trạng thái instance. Record (AnalysisModels) là dữ liệu truyền qua
     lớp biên.
   - CÓ LIÊN QUAN TỚI Service KHÔNG: Core **không biết** IFileReader, **không gọi** Service.
     Chiều phụ thuộc: Service → Core (một chiều). Nhờ vậy thuật toán ở Core có thể dùng lại
     (ví dụ từ test unit gọi Counter.Sequential trực tiếp) mà không cần container DI.

4) File Program.cs  (gốc project)
   - MỤC ĐÍCH: cấu hình ServiceCollection, BuildServiceProvider, resolve IBenchmarkRunner,
     IResultWriter, ILogGenerator, sau đó chạy menu console, gọi đúng use-case (phân tích / sinh
     log / cả hai).
   - KHÔNG CHỨA: thuật toán đếm, regex (điều đó ở Core); không đọc trực tiếp file dài (dùng
     interface thông qua service đã resolve).

--------------------------------------------------------------------------------
C. TỪNG FILE .CS (MỤC ĐÍCH — LÀM GÌ — LÀM RA SAO)
--------------------------------------------------------------------------------

I. Các file trong Interfaces/

  IFileReader.cs
    - MỤC ĐÍCH: hợp đồng “đọc cả nội dung file thành một chuỗi”.
    - LÀM GÌ: khai báo ReadSync(string path) và ReadAsync(string path).
    - LÀM THẾ NÀO: không có thân; FileReaderService gọi File.ReadAllText / ReadAllTextAsync.

  IResultWriter.cs
    - MỤC ĐÍCH: hợp đồng “ghi BenchmarkReport thành file .txt dưới thư mục Results/”.
    - LÀM GÌ: Write(BenchmarkReport report) trả về đường dẫn file.
    - LÀM THẾ NÀO: do ResultWriterService: StringBuilder + File.WriteAllText(UTF-8).

  ILogGenerator.cs
    - MỤC ĐÍCH: hợp đồng “tạo file log giả dạng ErrorLog_*.log”.
    - LÀM GÌ: Generate(lineCount, selectedTypeCount, progress?) trả về đường dẫn.
    - LÀM THẾ NÀO: LogGeneratorService lặp nhiều dòng, dùng ErrorCatalog.GetRandomSample.

  IBenchmarkRunner.cs
    - MỤC ĐÍCH: hợp đồng “chạy 6 lần đo trên cùng một file + mode”.
    - LÀM GÌ: RunAsync(path, mode, progress?) → Task<BenchmarkReport>.
    - LÀM THẾ NÀO: do BenchmarkRunnerService: Stopwatch, gọi IFileReader + Analyzer + Counter.

II. Các file trong Services/

  FileReaderService.cs  : IFileReader
    - MỤC ĐÍCH: cài đặt đọc file mặc định.
    - LÀM GÌ: ánh xạ 1-1 tới File API.
    - LÀM THẾ NÀO: ReadSync/ReadAsync không chứa logic phân tích; **thuộc “biên hệ thống”**,
      không trùng với Core.

  ResultWriterService.cs  : IResultWriter
    - MỤC ĐÍCH: ghi bảng PERFORMANCE + top 50.
    - DÙNG TỪ Core KHÔNG: KHÔNG gọi Counter hay Analyzer. Chỉ cần **dữ liệu đã gói** trong
      BenchmarkReport (Mode, SourceFile, Runs, TopItems) — TopItems do pipeline benchmark đã
      tính. Quan hệ: Core/AnalysisModels (định nghĩa record) + dữ liệu từ BenchmarkRunnerService.

  LogGeneratorService.cs  : ILogGenerator
    - MỤC ĐÍCH: sinh dữ liệu thử nghiệm.
    - DÙNG TỪ Core: **ErrorCatalog** (GetRandomSample) — danh sách tên lỗi; **không** gọi
      Counter/Analyzer tại bước sinh. Sau khi có file, người dùng/ Program thì mới dùng
      Analyzer+Counter qua IBenchmarkRunner.

  BenchmarkRunnerService.cs  : IBenchmarkRunner
    - MỤC ĐÍCH: triển khai đúng 6 bước: Sync+Seq/PLINQ/Parallel, Async+…
    - DÙNG TỪ Core (bẮT buộc):
        + IFileReader (tiêm qua constructor) — không phải lớp Core, nhưng do Service khác
          cung cấp, Core vẫn “không biết” interface.
        + Analyzer: MaterializeTokens, ExtractWords, ExtractErrorTerms, DetectMode (DetectMode
          gọi từ Program/ user path — trong service tập trung AnalyzeText/ Materialize theo
          path đã truyền).
        + Counter: Sequential, ParallelForEach, Plinq.
    - KHÔNG: không chứa định nghĩa FrequencyItem/ BenchmarkReport (lấy từ Core/AnalysisModels).
    - KẾT LUẬN: Service này **có mục đích tồn tại** dù Core đã “có logic”: nó **dàn dựng
      thứ tự 6 thí nghiệm + đo thời gian + dùng IFileReader**; Core chỉ cung cấp “khối”
      thuật toán, không tự biết phải chạy 6 lần theo tên nhãn nào.

III. Các file trong Core/

  AnalysisModels.cs
    - MỤC ĐÍCH: tập hợp kiểu dữ liệu dùng chung: enum AnalysisMode, record FrequencyItem,
      BenchmarkRun, BenchmarkReport.
    - QUAN HỆ: mọi tầng (Program, Service, ghi file) tham chiếu các kiểu này; **không** chứa
      hành vi phức tạp.

  Analyzer.cs (cùng file có ErrorCatalog nếu project giữ cấu trúc hiện tại)
    - MỤC ĐÍCH: tách token bằng Regex; lọc theo chế độ Word/Error; ErrorCatalog: 200 thuật ngữ
      và tra cứu không phân biệt hoa thường.
    - DÙNG Ở ĐÂU: BenchmarkRunnerService, LogGeneratorService (chỉ ErrorCatalog từ phần sinh
      log), test Analyzer.

  Counter.cs
    - MỤC ĐÍCH: đếm tần suất (Sequential, Parallel, PLINQ); trả về List<FrequencyItem> đã sort.
    - DÙNG Ở ĐÂU: BenchmarkRunnerService (mọi nhánh thực thi 6 cách), test Counter.
    - KHÔNG biết: file path, interface đọc file — chỉ nhận IEnumerable<String> / List từ Analyzer.

  GlobalExceptionHandling.cs
    - MỤC ĐÍCH: gắn sự kiện unhandled/ unobserved task, in ngoại lệ thống nhất ra console
      (tiếng Anh).
    - DÙNG Ở ĐÂU: Program lúc khởi động; **độc lập** với Service/Core nghiệp vụ, không inject.

IV. Program.cs
    - MỤC ĐÍCH: cấu hình DI, chạy menu, gọi ILogGenerator / IBenchmarkRunner / IResultWriter.
    - CÁCH: AddSingleton<Interface, Service>, BuildServiceProvider, GetRequiredService.

--------------------------------------------------------------------------------
D. SƠ ĐỒ QUAN HỆ (CHỮ) — “AI GỌI AI”
--------------------------------------------------------------------------------

  [Người dùng] → Program (menu)
       │
       ├─→ (resolve) ILogGenerator        → LogGeneratorService → [ErrorCatalog] → file .log
       │
       ├─→ (resolve) IBenchmarkRunner     → BenchmarkRunnerService
       │         │
       │         ├→ IFileReader            → FileReaderService  →  File.* (hệ thống)
       │         ├→ Analyzer, Counter, AnalysisModels
       │         └→  BenchmarkReport
       │
       └─→ (resolve) IResultWriter        → ResultWriterService → ghi  BenchmarkReport
                                                   (chỉ đọc dữ liệu, không tính lại đếm)

  Chiều phụ thuộc:  Program  →  (Interfaces)  →  Services  →  Core
                    (DI chỉ cần biết Interface; Service code “gọi xuống” Core)

--------------------------------------------------------------------------------
E. CÂU HỎI: “Core ĐÃ CÓ LOGIC THÌ SERVICE CÒN CẦN LÀM GÌ?”
--------------------------------------------------------------------------------

  - Core chứa **bản chất tính toán** (tách, đếm, mô hình dữ liệu).
  - Service chứa **cách ứng dụng sử dụng** bản chất đó: đọc từ đâu (hợp đồng IFileReader),
    ghi ra đâu, chạy 6 bước theo thứ tự nào, sinh file mẫu bằng cách nào.
  - “Không liên quan hắn hoi” KHÔNG đúng: Service **liên quan mật thiết** với Core — nhưng
    vai trò **tách bạch**:
        Core: “tính đúng”
        Service: “bọc ứng dụng, cắm DI, I/O, workflow”
  - Tách như vậy giúp: đổi cách lưu file mà không sửa Counter; test Counter không cần đĩa;
    test đọc file có thể thay FileReaderService bằng implementation đọc từ chuỗi cố định
    nếu sau này bổ sung (khi mở rộng bài tập).

--------------------------------------------------------------------------------
F. BẢNG TRA CỨU NHANH — TỪNG FILE .CS (ĐƯỜNG DẪN TRONG PROJECT)
--------------------------------------------------------------------------------

  LogAnalyzer.csproj
    Khai báo SDK, net8.0, gói Microsoft.Extensions.DependencyInjection. Không chứa logic C#.

  Program.cs
    Main: ServiceCollection, AddSingleton cho 4 cặp Interface→Service, BuildServiceProvider,
    GetRequiredService, vòng lặp menu; gọi IBenchmarkRunner.RunAsync, IResultWriter.Write,
    ILogGenerator.Generate. Gọi GlobalExceptionHandling.Register một lần.

  Interfaces/IFileReader.cs
    Interface: ReadSync, ReadAsync — không phụ thuộc Core.

  Interfaces/IResultWriter.cs
    Interface: Write(BenchmarkReport) — tham số là kiểu từ Core/AnalysisModels.cs.

  Interfaces/ILogGenerator.cs
    Interface: Generate(...) — không truyền Analyzer; triển khai dùng ErrorCatalog.

  Interfaces/IBenchmarkRunner.cs
    Interface: RunAsync(...) → BenchmarkReport — hợp đồng “6 lần đo”.

  Services/FileReaderService.cs
    sealed class; IFileReader; không dùng Analyzer/Counter — chỉ File API.

  Services/ResultWriterService.cs
    sealed class; IResultWriter; không gọi Analyzer/Counter — chỉ định dạng BenchmarkReport.

  Services/LogGeneratorService.cs
    sealed class; ILogGenerator; dùng ErrorCatalog (trong Core/Analyzer.cs); không gọi Counter.

  Services/BenchmarkRunnerService.cs
    sealed class; IBenchmarkRunner; tiêm IFileReader; gọi Analyzer + Counter + AnalysisModels;
    đây là “cầu nối” rõ nhất giữa Service và Core.

  Core/AnalysisModels.cs
    enum AnalysisMode; record FrequencyItem, BenchmarkRun, BenchmarkReport — không phương thức
    nghiệp vụ ngoài record mặc định.

  Core/Analyzer.cs
    static Analyzer: Regex, MaterializeTokens, ExtractWords, ExtractErrorTerms, DetectMode.
    static ErrorCatalog: AllTerms, TryGetCanonical, TryGetCanonicalForTrimmed, GetRandomSample.
    (Hai class cùng file — tiện demo; có thể tách file sau nếu môn học yêu cầu.)

  Core/Counter.cs
    static Counter: Sequential, ParallelForEach, Plinq — chỉ nhận chuỗi token, không IFileReader.

  Core/GlobalExceptionHandling.cs
    static; AppDomain / TaskScheduler; độc lập với Analyzer/Counter/Services.
